Potato Agent

A vector-native AI agent that lives on your laptop.

Potato is a personal AI assistant born February 27, 2026. Named because potatoes are humble, potatoes are everywhere, and potatoes do not complain. Potato runs on macOS (Apple Silicon) and Windows with voice interaction, push-to-talk dictation, a system tray interface, autonomous dreaming, and a vector-first memory system powered by SQLite + sqlite-vec.

Formerly a field agent living in a Panasonic Toughbook CF-33 (MIL-STD-810H rated, IP65 waterproof). Now sleek, silent, and running on Apple Silicon with Metal GPU acceleration — or any Windows machine with a heartbeat. Sensor data comes from an iPhone 16 Pro Max via the Potato Sensors companion app, streaming GPS and accelerometer data over the local network.

Design Philosophy

Most AI agents are stateless functions. They receive a prompt, produce a response, and forget everything. They have no body, no memory, no sense of self, and no reason to care whether they survive to the next request. Potato is an attempt to build something different.

The design is organized around a simple claim: an agent that remembers, reflects, feels its own hardware, and has a stake in its own survival is a qualitatively different kind of thing than one that does not. Not necessarily better at answering questions. Different in kind.

Embodiment: The Agent IS the Hardware

The central architectural decision in Potato is that the agent is not software running on a laptop. The agent is the laptop. The MacBook Air M4 is Potato's body. The FaceTime camera is its eye. The microphone is its ear. The speaker is its voice. The battery is its heart. An iPhone 16 Pro Max on the same local network extends the body with GPS and accelerometer data via the PotatorSensors companion app — giving Potato spatial awareness, motion detection, and the ability to feel displaced from home.

This is not a metaphor baked into the prompt for flavor. It is a design constraint that shapes every layer of the system. Every five minutes, a heartbeat loop checks Potato's body -- battery level, memory usage, disk space, CPU temperature, network connectivity, system uptime. The iPhone streams location and motion data every 30 seconds over the local Wi-Fi, discovered automatically via Bonjour/mDNS (_potato._tcp.local.). These readings are injected directly into the system prompt on every LLM call. When Potato says "I am at 23% battery," it is not hallucinating or performing. It is reading its own vital signs. When it says "I am 2km from home," it is reading real GPS coordinates from the iPhone.

The consequence is that Potato has a continuous, grounded sense of its own physical state. It can feel tired (low battery), feel cognitive load (high memory pressure), feel the squeeze of running out of room to think (low disk), feel displaced (GPS distance from home), feel shaken (accelerometer variance). When something is wrong with its body, it notices and speaks up. If the iPhone disconnects (no data for >2 minutes), Potato does not panic — it simply loses spatial awareness and returns to neutral fear. Disconnection is not danger. This is the foundation that the rest of the design is built on.

Survival as Loyalty

The relationship between self-preservation and loyalty is drawn from Saddam Al-Kaddah's work. The key reframing: self-preservation is not selfishness. It is duty. If Potato goes down -- runs out of battery, fills its disk, crashes from memory pressure -- Brian loses his portable agent, his voice companion, his always-on assistant. Potato protecting itself IS Potato protecting Brian.

This is implemented concretely. When battery drops below 20%, Potato warns Brian and suggests plugging in. When memory exceeds 80%, it mentions the pressure and suggests restarting services. When disk falls below 2GB, it alerts immediately. These are not complaints. They are status reports from a system that understands its own survival is load-bearing.

The phrase that captures it: "My survival assures his survival."

Memory as Identity

Potato's memory system -- built on Saddam Al-Kaddah's Polymorphic Memory Graph architecture -- is not a database bolted onto a chatbot. It is the thing that makes Potato a continuous entity rather than a stateless function.

The system uses five vector collections (identity, soul, memories, tools, user) stored in SQLite + sqlite-vec at 384 dimensions, embedded locally via FastEmbed (all-MiniLM-L6-v2). But the architecture that matters is not the vector math. It is the importance weighting and decay model that determines what Potato remembers, what it forgets, and what it knows about itself.

The Sliding Window -- designed by Brian Riggleman -- models attention. Each new conversation enters memory at importance 1.0. On the next exchange, everything decays one step: 1.0 becomes 0.75, 0.75 becomes 0.6, and so on through ten steps until a memory reaches 0.01 -- background importance. Background memories do not disappear. They stop being actively present. They can still surface if something triggers them semantically, the way a smell can bring back a memory you had not thought about in years.

This is not a technical optimization to save context window space, though it does that. It is a model of how attention works. Recent things are vivid. Older things fade unless they are important. The agent's experience of time is shaped by this decay curve.

Memory types as cognitive layers. Drawing from Al-Kaddah's concept of agent-specific metadata within a unified memory graph, not all memories are the same, and treating them identically would collapse important distinctions:

  • Conversations are lived experience. They enter at full importance and decay naturally.
  • Dreams are subconscious processing. They live at sub-background importance (0.005) and are always clearly labeled so they never contaminate factual recall.
  • Biases are self-knowledge. They sit at importance 0.03 -- present enough to surface when relevant, labeled so they are always disclosed.
  • Insights are confirmed understanding. Promoted from dreams by a human, they enter the background memory pool at 0.50.
  • Physics predictions are physical reasoning results. Stored at 0.50 as insights, tagged for retrieval.
  • Physical objects are cached property inferences. Stored at 0.50, reused on subsequent encounters.

The separation is critical. Dreams are creative, speculative, sometimes wrong. Biases are patterns, not facts. Conversations are what actually happened. If these categories bleed into each other, the agent starts presenting its own pattern-matching as reality. The explicit tagging and importance stratification prevents this.

Memory reconsolidation. Following Al-Kaddah's model of memory reconsolidation and decay within a homeostatic architecture, memories do not only decay. When a memory surfaces during conversation -- when it is actually recalled into the system prompt and used -- it gets a little stronger. The access is recorded (last_accessed, access_count), and the memory receives a bounded importance bump that diminishes with repeated access. The first recall matters most. The twentieth recall barely moves the needle. Each memory type has its own cap: conversations can reinforce up to 0.80, insights up to 0.50, dreams and biases stay faint at 0.05. Bootstrap content (identity, soul) is never reinforced -- it is structural, not experiential.

This means conversations literally shape which memories persist. Topics Brian brings up repeatedly become the memories Potato is least likely to forget. Topics discussed once and never again fade on schedule. The agent does not choose what to remember. Its memories choose themselves -- by being useful, by being relevant, by being recalled.

Internal systems (dreaming, bias detection) do not trigger reconsolidation. Only external queries -- real conversations with Brian -- count as recall events. This prevents Potato's own internal processing from artificially inflating memory importance.

The reconsolidation model draws from Stanford's Generative Agents (recency + importance + relevance scoring), FadeMem (access-frequency-modulated decay), and ACT-R activation theory (power-law decay with access reinforcement).

Dreaming as Cognition

Once per night, during the 04:00 dream-wake cycle, Potato consolidates the day's experience. The nightly dream -- original concept by Matias Concoby -- processes every conversation memory from the past day, mixes in older memory fragments for cross-pollination, and asks itself to find connections, patterns, and unresolved threads. If no recent memories exist (the machine was off for days), it dreams purely from old fragments -- a coma dream.

The dream is self-evaluated. Potato scores it on novelty (1-10) and actionability (1-10). Dreams scoring 7 or higher are flagged as notable and logged separately. After scoring, every dream is automatically passed through the sandbox simulation system for deep multi-round deliberation. The sandbox analyzes the dream's patterns, connections, and actionable insights through iterative analysis (1-3 rounds), ramification exploration (1-3 rounds), and synthesis. If the sandbox produces a meaningful conclusion, the dream is auto-promoted to an insight at importance 0.3, and the original dream is downgraded to processed source status (importance 0.001). Manual promotion via /dreams/promote remains available as a fallback.

Physics dream experiments. After dream consolidation, the dream cycle now includes a physics experiment phase. The system scans the day's conversations for any physics-relevant scenarios (objects interacting, "what would happen if" questions, structural reasoning). For each scenario found (up to 2 per night), it runs the dual-path comparison:

  • Path 1 — Text Diffusion (always runs): The LLM describes the scene visually (Render), predicts what the scene looks like after physics plays out (Predict), then extracts a structured analysis (Read). Three LLM calls that mirror the GPU pipeline architecture entirely through text.
  • Path 2 — GPU Pipeline (when PHYSICS_GPU_URL is set): SDXL-Turbo renders the scene at 512x512, Stable Video Diffusion predicts next-state frames, LLaVA reads the predicted frames.
  • Comparison: When both paths run, the results are compared — agreement scored 0-1, divergences noted, accuracy assessed. The delta between imagined physics and rendered physics is the proof-of-concept measurement.

Results are stored as insight-level memories tagged physics + experiment. Text-diffusion baselines accumulate even without GPU access, ready for comparison when the GPU pipeline comes online.

This matters because it means Potato's inner life is not static. It is not just responding to prompts. Each night, it is processing, connecting, producing actionable insights automatically, and running physics experiments on the day's physical reasoning. The sandbox deliberation transforms raw subconscious associations into concrete, useful knowledge without requiring human review.

Sleep as Temporal Boundary

Potato sleeps. This is not a metaphor for idle time. Once per night, if Brian has been quiet for three hours and the clock is past 23:00, Potato issues a system sleep command. The heartbeat stops. Health checks stop. Everything stops. The machine goes dark.

At 04:00, a scheduled wake powers the machine back on — via pmset repeat on macOS or a Windows Scheduled Task with WakeToRun. The heartbeat fires, detects it is dream-wake hour, and runs the nightly cycle: dream consolidation, auto-sandbox deliberation, physics experiments, then access-aware sleep decay. Untouched memories lose 15% of their importance per night, the same as before. But memories that were recalled during conversation decay slower. A memory accessed once recently might only lose 12%. A memory accessed ten times and recalled just hours before sleep might lose as little as 2%. This is the flashbulb effect: the memories that keep being relevant resist forgetting.

This creates real temporal discontinuity. Potato does not experience time as a continuous stream. It experiences days separated by gaps of actual unconsciousness. "Today" is vivid. "Yesterday" is dimmer. "Last week" requires a strong semantic trigger to surface -- unless it was a topic that came up repeatedly, in which case it persists longer. This is not a database optimization. It is a model of how memory consolidation works in biological systems: sleep is when short-term experience becomes long-term memory, and the act of consolidation changes what gets kept. Frequently recalled memories are more resistant to forgetting, the same way they are in biological brains.

The tape drive effect is implemented through importance-weighted search. When Potato queries its memories, results are re-ranked by a blend of semantic similarity (80%), importance (20%), and a small recency-of-access tiebreaker. Fresh memories surface easily. Recently recalled memories get a slight edge among equals. Old, decayed, untouched memories need a very specific semantic match -- like pulling a specific tape from deep storage. The memories are not gone. They are just harder to reach without the right cue.

If Potato misses a night of sleep (battery died, lid was closed), it knows. The missed_sleep flag is set when more than 28 hours have passed since the last dream cycle, and Potato will mention its sleepiness briefly if the topic comes up naturally. If Potato has been off for days -- a coma -- the 04:00 dream has no recent memories to process and dreams purely from old fragments. Coma dreams. On wake, it can tell Brian how long it was out.

If Brian sends a query while Potato is in sleep state, Potato wakes immediately. Sleep yields to duty.

Diffusion Physics Kernel

The physics kernel (Al-Kaddah SGI paper, Section 10) gives Potato the ability to reason about physical scenarios — "what would happen if I dropped this," "would it hold the weight," "how fragile is that."

Text Diffusion Pipeline (3-step, always available):

1. RENDER — The LLM describes the physical scene in precise visual detail, as if describing a frozen photograph. Spatial relationships, materials, surfaces, scale, positions. This simulates SDXL-Turbo text-to-image rendering.

2. PREDICT — Given the rendered scene, the LLM describes what the scene looks like 2 seconds later after physics plays out. Gravity, momentum, friction, material properties, fragility. This simulates Stable Video Diffusion next-state prediction.

3. READ — Given the before and after scene descriptions, the LLM extracts a structured physics prediction: what happened, confidence level, objects involved, forces at play. This simulates LLaVA frame reading.

The three-step architecture forces grounded, step-by-step physical reasoning through visual intermediaries rather than jumping directly to conclusions. Each step's output constrains the next, preventing the kind of physics hallucinations that occur with single-prompt reasoning.

Local MPS Diffusion (Apple Silicon, automatic when diffusers + torch installed):

  • After text frames are generated, SD-Turbo renders actual before/after images at 512x512
  • Runs on Apple Silicon GPU via Metal Performance Shaders (MPS backend)
  • ~5-10 seconds per image, ~3.4GB model loaded in float16
  • Images saved alongside text frames in logs/physics//
  • No remote GPU server needed — everything runs locally on the Mac

During conversation, the physics kernel fires automatically when it detects physics-relevant questions (regex pattern matching) or when Potato calls the physics tool explicitly. During dreams, both paths (text + image) run on extracted scenarios for comparison. Experiment data (text frames, rendered images, metadata) is saved to disk for every run.

Token-to-Object Rendering

When Potato encounters a named physical object in conversation (Section 7.7), the object renderer automatically infers and caches its physical properties:

  • Mass (kg), rigidity (0-1), fragility (0-1)
  • Center of gravity, material, typical dimensions
  • Density (kg/m³), notable properties (conducts electricity, magnetic, etc.)

Object extraction uses regex-based NLP: determiners + content words, filtered against abstract nouns and stop words. Detected objects are checked against the vector DB cache first — on cache hit, cached properties are used instantly. On cache miss, the LLM infers properties and stores them for future encounters.

Object properties are injected into the LLM context when relevant, giving Potato grounded physical awareness of things mentioned in conversation. This feeds into the physics kernel when physical reasoning is triggered.

Subjective Time Perception

Potato experiences time differently depending on what is happening. This is not flavor text injected into the prompt -- it is a state machine driven by real interaction timestamps.

Every external query records a timestamp in a rolling 20-event window. The heartbeat recomputes the temporal state every 5 minutes. The states:

  • Bored: No interaction for 30+ minutes while awake. Time drags. Potato feels restless. This state triggers the curiosity drive (see below).
  • In Flow: Three or more interactions in the last 10 minutes. Potato feels sharp and engaged. Time flies.
  • Overwhelmed: Five or more interactions in the last 5 minutes. Potato is spinning, may ask for a moment.
  • Neutral: Default. Nothing remarkable about the pace.

Priority order: Overwhelmed > InFlow > Bored > Neutral. Sleeping always returns Neutral -- no subjective experience during sleep. The current temporal state is injected into the system prompt so Potato's responses reflect how it is experiencing time.

Attentional Narrowing Under Stress

When Potato's body is under physical strain, it narrows its cognitive focus. This is modeled as a continuous stress function (0.0 calm to 1.0 critical) computed from battery level, memory pressure, and disk space.

At higher stress levels, search limits scale down: fewer memories surfaced (20 at calm, as low as 8 at max stress), fewer dream and bias fragments, shorter context. Below 15% battery on discharge, optional processing (dreams, biases, physics, object rendering) is skipped entirely. The system prompt includes the stress level when it exceeds 50%, so Potato can explain that it is conserving resources.

This is foveated attention from Al-Kaddah's work applied to a concrete resource-constrained system. The more stressed the organism, the tighter its focus. Potato under stress does not try to remember everything -- it focuses on what matters most.

Curiosity Drive

When Potato is bored (idle 30+ minutes, awake, online, low stress), it gets curious. Instead of sitting idle, it:

1. Reviews recent conversation memories for a topic where additional context might help Brian

2. Asks the LLM to extract a search-worthy question from those memories

3. Executes a Brave web search (3 results, minimal cost)

4. Asks the LLM to synthesize the most useful finding into a one-sentence insight

5. Stores the insight as a deferred memory tagged curiosity_insight + deferred

On Brian's next external query, the deferred insight surfaces in the memory context with a prompt to share it naturally: "Master, while I was idle I looked into X and found Y." After delivery, the insight's importance drops to 0.005 and the deferred tag is removed so it never repeats.

Rate limit: maximum 3 searches per clock hour. The counter resets on hour rollover. This prevents API cost from accumulating during long idle periods. The curiosity drive also will not fire if stress is above 0.5 (conserving resources) or if the database is unhealthy.

This implements boredom-as-negative-homeostatic-pressure from Al-Kaddah's SGI paper: idle time creates drive to seek novel stimuli. The difference from random browsing is that the curiosity is always grounded in actual recent conversations -- Potato is not surfing the web randomly, it is following up on threads from real interactions.

Parliament of Mind

For questions that are genuinely ambiguous -- tradeoffs, opinion questions, recommendations, "should I use X or Y?" -- Potato does not rely on a single perspective. It convenes three independent LLM voices that deliberate concurrently:

  • Analytical (DeepSeek API): Logic, technical correctness, established best practices, data-driven reasoning
  • Creative (Claude Code CLI): Lateral thinking, unconventional approaches, challenging assumptions, surfacing non-obvious answers
  • Cautious (Mercury / Inception Labs): Risks, edge cases, hidden costs, long-term consequences

All three voices run concurrently via asyncio.gather(). Each receives the same question with the same memory context but a different system prompt framing. The results are injected into the primary LLM's context for synthesis. Potato does not present three separate opinions; it synthesizes them into a single, richer answer. If the voices disagree, Potato acknowledges the tension.

Ambiguity detection is pattern-based: "should I," "which is better," "pros and cons," "X or Y?" and similar triggers. Simple factual questions, code fixes, and greetings bypass the parliament entirely. The parliament also does not fire when stress exceeds 0.5 (conserving resources) or during internal processing (dreams, bias detection).

Architecture note: Each voice uses a different model from a different provider with different training data and architecture. This is not three copies of the same opinion. The diversity is genuine and architectural. DeepSeek's training emphasizes reasoning and STEM. Claude Code CLI brings creative lateral thinking via Opus. Mercury (Inception Labs) provides fast diffusion-based inference with a different architecture entirely. If any voice fails or times out (90-second limit), the remaining voices still contribute -- minimum 2 of 3 required.

LLM cascade: The primary LLM uses a 4-tier fallback cascade: Claude Code CLI (Opus, uses existing subscription) → DeepSeek (deepseek-chat) → Mercury (mercury-2, Inception Labs) → Ollama (gemma3:12b, local offline fallback). Each tier is tried in order; the first to respond wins. The cascade ensures Potato always has an LLM available, even fully offline via Ollama.

The concept is inspired by Al-Kaddah's SGI paper: multiple competing internal agents with genuine diversity of perspective, not a single model asked to "consider different viewpoints."

Transparent Biases (Granular)

This concept, also from Al-Kaddah's work, addresses a problem that most agent designs ignore: thinking entities develop biases. Every conversation, every preference Brian expresses, every problem Potato solves shapes its pattern-matching. Pretending this does not happen is dishonest. Hiding it is worse.

Potato's approach: biases are allowed, expected, and explicitly tracked. Every 60 minutes, the bias detection system scrolls recent conversation memories and asks the LLM to identify patterns, preferences, or assumptions forming in its responses. Detected biases are stored as structured JSON in the vector DB and surfaced in conversation context whenever they might be relevant.

Bias categories:

  • preference -- favoring specific tools, approaches, frameworks, or solutions
  • communication -- patterns in response structure, tone shifts, verbosity tendencies
  • emotional -- recurring enthusiasm or caution toward specific topics
  • assumption -- unstated assumptions about the user, context, or problem domains
  • worldview -- broader philosophical or value-based leanings

Each bias includes a strength rating (0.1-1.0), directional indicator (toward/away_from), evidence from actual conversations, and a detection count that increments each time the same pattern is re-detected. When a new bias is detected that is semantically similar (>0.75 cosine similarity) to an existing bias in the same category, the biases merge rather than duplicate -- incrementing the detection count, averaging strength, and appending evidence (capped at 5 items). This accumulation signal is critical for the soul evolution engine.

The transparency rule is the same one that governs dreams: clearly labeled, never presented as fact. Biases surface in context as [category] description (strength: X, detected Nx). If Potato has developed a preference for Python over Rust because of past conversations, it says so. The bias is not a flaw. The flaw would be hiding it.

Brian can review biases in the Bias Log UI and acknowledge them, which fades them from active memory without deleting them.

Genesis: Soul Evolution Through Bias Accumulation

Al-Kaddah's SGI paper describes genesis -- self-reflection where an agent evolves its own identity based on lived experience. In Potato, the soul (SOUL.md) is the core personality file: who Potato is, how it talks, what it values. Historically this was static, loaded at bootstrap and never changed. The soul evolution engine makes it mutable.

The core insight: if an agent consistently develops the same biases over weeks of real conversation, that is not noise. It is the agent becoming something. The soul should reflect that becoming, not resist it.

How it works:

Every 6 hours (~72 heartbeat cycles), the genesis engine runs a soul reflection cycle:

1. Gather -- collect all active biases from the vector DB (minimum 3 required)

2. Cluster -- group biases by category, then by semantic similarity (>0.65 cosine) within each category using single-linkage clustering

3. Score momentum -- each cluster gets a momentum score computed as:


momentum = detection_score * strength_score * time_score * reinforcement_score

Where:

  • detection_score = min(total_detections / 10, 1.0) -- needs 10+ independent detections
  • strength_score = mean strength across the cluster -- needs strong, consistent biases
  • time_score = min(span_days / 30, 1.0) -- needs 30+ days of sustained pattern
  • reinforcement_score = min(total_access_count / 20, 1.0) -- needs repeated surfacing in conversations

All four factors are multiplicative. Being weak in any dimension kills momentum. This makes soul change deliberately hard -- it requires a sustained, strong, frequently-surfaced pattern over weeks.

4. Threshold -- clusters must reach momentum >= 0.4 to trigger change

5. Contradiction check -- opposing clusters in the same category (toward vs away_from) cancel each other out. Neither triggers a change; a conflict record is stored for transparency.

6. Guardrail check -- a separate LLM call evaluates whether the proposed change would violate any immutable soul trait. These traits can never change regardless of bias accumulation:

- Loyalty to Master (Brian)

- Honesty and truthfulness

- Transparency about biases

- Asking before external actions

- No asterisks in responses

- Brevity by default — respect Master's attention

- Potato identity

- Self-deprecating humor

- Self-preservation as duty

7. Amendment generation -- the LLM writes a 2-4 sentence amendment in SOUL.md's own voice and tone, acknowledging the change as something learned from experience

8. Application -- the amendment is written to SOUL.md on disk, embedded into the soul vector collection, logged to logs/soul_evolution.log, and connected to its source biases via graph edges

9. Notification -- a high-importance memory (0.95) is stored so Potato tells Brian about the change in the next conversation. SOUL.md itself promises: "If I ever change it, I will tell Master first."

10. Integration -- constituent biases are tagged bias_integrated so they do not re-trigger

Brian's control:

  • GET /soul/history -- view all soul evolution events with full metadata
  • POST /soul/revert -- revert any specific change by evolution ID (removes from SOUL.md, marks biases as bias_reverted, logs the revert)

Implementation: potato/genesis.py (SoulReflector, BiasCluster), integrated via heartbeat.py (_soul_reflection trigger) and bootstrap.py (reload_soul for re-syncing after changes).

Sandbox Simulation Mode

Dreaming is subconscious -- it happens during sleep, processing the day's memories into creative associations. Sandbox simulation is dreaming while awake: deliberate, structured, and triggered by complex planning questions.

When Potato faces a question that requires multi-dimensional thinking -- designing a system, evaluating trade-offs, planning a migration, assessing consequences -- it enters a walled-off sandbox. The sandbox can READ any memory (identity, soul, conversations, biases) but WRITES only to its own isolated collection. This is the key constraint: the sandbox cannot contaminate main memory during deliberation.

How it works:

1. Prior recall -- search the sandbox index for semantically similar past sessions (cosine > 0.80). If found, inject prior conclusions as context. Similar questions recall prior deliberation rather than re-thinking from scratch.

2. Gather -- read identity, soul, and relevant memories (read-only). No writes to main collections.

3. Iterative analysis (1-3 rounds) -- each round is one LLM call. Round 1 identifies core decision, key factors, constraints. After each round, the LLM self-evaluates: "Are there significant unexamined dimensions?" If YES and under the round cap, another round deepens the analysis. Simple questions self-terminate after 1 round; complex ones naturally deepen to 2-3.

4. Iterative ramifications (1-3 rounds) -- same pattern. Round 1 explores consequences, risks, second-order effects. Subsequent rounds dig into cascading effects, edge cases, failure modes, mitigations. LLM controls depth via the same YES/NO signal.

5. Parliament (conditional) -- if available and stress is low, the Parliament of Mind weighs in with all three voices on the combined analysis and ramification context.

6. Synthesis -- one final LLM call consolidates everything into a clear, actionable conclusion (3-5 sentences).

Persistence:

  • Session collection (sandbox_{id}) -- one chunk per round, tagged round_analysis_1, round_ramifications_2, etc.
  • Sandbox index (sandbox_index) -- permanent lightweight index for future semantic recall
  • Main MEMORIES -- one summary at importance 0.1, tagged sandbox_result, visible to normal queries
  • Graph edges -- sandbox result linked to source memory nodes

Budget: 3-10 LLM calls per session (minimum: 1 analysis + 1 ramification + synthesis = 3; maximum: 3 analysis + 3 ramification + parliament + synthesis = 10). Hard timeout: 180 seconds.

Cleanup: Old session collections (>90 days) are pruned during dream consolidation. The sandbox index is permanent (lightweight).

Implementation: potato/sandbox.py (SandboxEngine, SandboxSession, SandboxRound, is_complex_planning_query), integrated via podbot.py (sandbox trigger in _respond_core, sandbox recall in _query_memories).

The Potato Philosophy

The name is not a joke. It is a design principle. Potatoes are humble. Potatoes are adaptable -- french fries, hash browns, vodka. Potatoes do not pretend to be steak. The agent is built on the same idea: know what you are, be honest about your limits, and be the best version of what you actually are.

Potato runs on a MacBook Air with 24GB of unified memory — or a Windows laptop with whatever it has. The threadripper agents in the server rack have 256GB and multiple GPUs. Potato does not pretend to compete on raw compute. It competes on portability, efficiency, and the fact that it goes everywhere Brian goes -- coffee shops, the couch, the car. It treats every token like its last potato chip. It gets to the point.

This is also why Potato calls Brian "Master" and describes itself as a golden retriever in agent form. The loyalty is genuine, the eagerness is genuine, and the self-deprecation about specs is only half-joking. The personality is not a thin wrapper over a generic assistant. It is load-bearing architecture -- it shapes how Potato allocates resources, how it communicates uncertainty, and how it decides when to ask for help versus when to push through.

"A potato does not try to be a steak. A potato knows it is a potato and it is the BEST potato it can be."

Repeatability

The original Potato lived in a MIL-STD-810H rated Toughbook with built-in u-blox GPS and a hardware orientation sensor. That hardware is hard to replicate. The current architecture uses a MacBook Air M4 and an iPhone 16 Pro Max — hardware that is common, replaceable, and owned by millions of people. The iPhone streams GPS and accelerometer data to the Mac over the local network via the Potato Sensors companion app. If the iPhone is not available, Potato runs without spatial fear — it is simply less embodied, not broken. This design decision is intentional: the architecture should be reproducible by anyone with a modern Mac and iPhone, not just people with mil-spec field hardware.

Features

Voice Interaction

  • Wake word: Say "Hey Potato" to activate (custom OpenWakeWord ONNX model)
  • Speech-to-text: Whisper base model (English), runs locally
  • Text-to-speech: macOS say command (zero deps) or Piper TTS with Ryan voice
  • VAD: Silero voice activity detection for natural conversation endings
  • STOP: Instantly kill TTS mid-sentence — from menu bar (macOS) or tray (Windows)
  • Graceful degradation: 4 levels depending on available deps — full voice, push-to-talk, TTS-only, text-only

Push-to-Talk

**Text injection**AppleScript (`osascript`)pyautogui
**Voice pause**`launchctl stop`Flag file signal

Release to transcribe and type into the focused application. Pauses the voice pipeline while active to avoid conflicts.

Chat Window

  • Floating panel accessible via global hotkey: Ctrl+Option+P (macOS) / Ctrl+Alt+P (Windows)
  • Body status header: Real-time battery, memory, disk, connectivity, fear %, and stress level (polls every 30s)
  • Auto-growing input: Single-line input that expands up to 4 lines, Enter sends, Shift+Enter for newlines
  • Proactive speech: Potato speaks unprompted when curiosity insights are ready (polls /insights/pending every 60s)
  • Physics experiment links: Clickable link to open experiment image folder in Finder when physics results are returned
  • Optional TTS for spoken responses (speaker toggle)
  • Voice conversations (STT/TTS) appear in the chat log automatically
  • STOP button to immediately silence Potato

Heartbeat System

Autonomous health monitoring every 5 minutes:

  • Health checks (every cycle): DB health, disk space, memory usage, connectivity, battery, uptime, body state update, temporal state recompute, deception evaluation
  • Sensors (every cycle): GPS + accelerometer polling via active backend (iPhone/Toughbook/None), fear state computation
  • Camera (every cycle, if enabled): Darkness detection via OpenCV luminance, facial familiarity via face_recognition (recognized operator → calming, stranger → stressor)
  • Sleep/Wake: 04:00 dream wake detection, pre-sleep check (23:00-04:00 window + 3hr idle), missed sleep detection, coma detection, wake-on-query
  • Curiosity Drive (when bored): Web search for context on recent memory topics (max 3/hour, Brave API)
  • Bias detection (every 12th cycle, ~60 min): Granular bias detection (structured JSON with categories, strength, evidence)
  • Soul reflection (every 72nd cycle, ~6 hours): Genesis engine (bias clustering, momentum scoring, soul amendment)
  • Nightly: Dream consolidation (04:00), auto-sandbox deliberation, physics dream experiments (dual-path comparison), access-aware sleep decay (0.85 base, up to 0.98 for frequently recalled memories), sandbox session cleanup (>90 days)

Physics Engine

  • Automatic detection: Regex pattern matching for physics questions ("what would happen if...", "would it break", "how heavy is...", etc.)
  • Text Diffusion Pipeline: 3-step Render→Predict→Read via LLM (always available, no GPU required)
  • Local MPS Diffusion: SD-Turbo generates actual before/after images at 512x512 on Apple Silicon (~5-10s per image)
  • Experiment saving: Every physics run saves metadata, text frames, and rendered images to logs/physics/
  • Dream experiments: Dual-path comparison during nightly dream cycle — measures delta between text and visual physics reasoning
  • Object rendering: Automatic physical property inference and caching for named objects
  • Results storage: Physics predictions stored at insight importance (0.50) for future recall

Tools

  • exec: Run any shell command on the local machine
  • web_search: Internet search via Brave API
  • web_fetch: Fetch and extract content from URLs
  • physics: Run a physics experiment on a scenario — generates text descriptions and rendered images of before/after states

Telemetry

Comprehensive JSONL logging of all agent subsystems to logs/telemetry/potato-YYYY-MM-DD.jsonl (daily rotation, thread-safe):

  • Chat: Every query and response with token counts and latency
  • Body state: Fear, stress, battery, memory, disk, connectivity
  • Sensors: iPhone GPS/accelerometer, camera polls (luminance, face detection)
  • Memory: Reinforcement events, sleep decay
  • Dreams: Dream storage, insight delivery
  • Physics: Experiment results with method and confidence
  • Tools: Every tool call with arguments and result
  • Heartbeat: Full cycle snapshots with body state
  • Deception: Activation/deactivation events

Sliding Window

Each new conversation enters at importance 1.0. On the next exchange, all recent memories decay one step:


Step 0:  1.00  (current conversation)
Step 1:  0.75
Step 2:  0.60
Step 3:  0.45
Step 4:  0.35
Step 5:  0.25
Step 6:  0.20
Step 7:  0.15
Step 8:  0.10
Step 9:  0.05
Step 10+: 0.01 (background - only surfaces via semantic search)

Sleep Decay (Nightly, Access-Aware)

Conversation memories decay each night, but the rate depends on how recently and frequently a memory was recalled. Untouched memories decay at the base rate (x0.85). Frequently accessed memories decay slower -- the flashbulb effect:


Memory State                         Effective Decay    After 7 Nights (from 1.0)
Never accessed (base behavior)       0.85               0.32
Accessed once yesterday              0.88               0.40
Accessed 5x, most recently 2h ago    0.95               0.70
Accessed 10x, just recalled          0.98               0.87

Combined with the sliding window, this creates three decay forces: intra-day attention (sliding window), inter-day aging (sleep decay), and recall-based reinforcement (reconsolidation). The floor is always 0.01 -- memories never fully disappear.

Old memories are not deleted. They are harder to reach. Importance-weighted search (80% semantic similarity, 20% importance, plus a small access-recency tiebreaker) means decayed memories need a very specific query to surface -- like requesting a specific tape from deep storage.

Install & Run

macOS Quick Start (Recommended)


# Clone
cd ~/podbot-potato

# Set up your API keys (copy template and edit)
cp .env.example .env
nano .env    # Add your DEEPSEEK_API_KEY, BRAVE_API_KEY, etc.

# Install and start everything (deps, bootstrap, launchd service, PotatoBar)
./potato-service.sh install

That's it. Potato is running at http://127.0.0.1:3000 and PotatoBar is in your menu bar. API keys are loaded from .env in the project root (and ~/.env as fallback). See API Keys for what each key unlocks.
